📊 连续分配管理方式总览

连续分配是指为每个用户程序分配一段连续的内存空间。根据分区策略不同，分为四大类：

分配方式 分区数量 分区大小 碎片类型 并发度 408考频
单一连续分配 1个用户区 固定（全部） 内部碎片 单道 ⭐⭐
固定分区分配 多个 固定 内部碎片 多道 ⭐⭐⭐
动态分区分配 多个 动态变化 外部碎片 多道 ⭐⭐⭐⭐⭐
动态重定位分区 多个 动态变化 外部碎片（可紧凑消除） 多道 ⭐⭐⭐

🎯 核心考点速记

  • 内部碎片：分区内部未用空间（单一连续、固定分区、分页）
  • 外部碎片：分区之间无法利用的小空闲块（动态分区、分段）
  • 四种顺序搜索算法：FF、BF、WF、NF 的查找策略和优缺点
  • 三种索引搜索算法：快速适应、伙伴系统、哈希算法
  • 紧凑技术：消除外部碎片，需要重定位寄存器硬件支持

1️⃣ 单一连续分配（Single Contiguous Allocation）

📐 内存结构

内存 = OS区 + 用户区

操作系统(OS)
用户程序（独占全部）

低地址 ←————————————————→ 高地址

📖 基本原理

  • 内存分为两部分：低地址区存放OS，高地址区存放用户程序
  • 用户程序独占整个用户内存空间
  • 任何时刻内存中只有一道用户程序

✅ ❌ 特点

优点

  • 实现简单，无需复杂硬件
  • 管理开销极小

缺点

  • 只能单道运行，资源利用率极低
  • 有内部碎片（用户程序 < 用户区时）
考研要点：不需要内存保护机构（因为只有一道程序），但现代实现中通常用界地址寄存器防止越界。

2️⃣ 固定分区分配（Fixed Partition Allocation）

📐 内存结构

等大小分区 vs 不等大小分区

每个分区只能装入一道作业

📖 基本原理

  • 系统初始化时，将用户内存划分为若干个固定大小的分区
  • 每个分区只能装入一道作业
  • 通过分区说明表管理各分区状态

📋 分区说明表

分区号 起始地址 大小 状态
1100K32K已分配

3164K64K已分配

🔀 两种划分方式

等大小分区

  • 所有分区大小相同
  • 程序太小 → 内部碎片大
  • 程序太大 → 无法装入

不等大小分区

  • 分区大小按需求预先设定
  • 灵活性稍好
  • 但仍可能产生内部碎片

🎯 关键结论

固定分区只有内部碎片，没有外部碎片！因为分区大小固定，不存在"总空闲足够但分散无法分配"的情况。

📖 基本原理

  • 不预先划分分区，作业装入时根据实际需要动态建立分区
  • 分区大小 = 作业实际需要的大小
  • 使用空闲分区表或空闲分区链记录未分配区域
分配过程：检索空闲分区 → 找出满足要求的分区 → 若分区>需求则分割（一部分分配，剩余仍为空闲）→ 若相等则直接分配

🔍 基于顺序搜索的四种算法

🥇 首次适应 FF

排序：按地址递增

策略：从链首开始，第一个满足的分区即分配

优点：简单、快；保留高地址大分区

缺点：低地址碎片多

408最常考

🥈 最佳适应 BF

排序：按容量递增

策略：找最小的满足分区

优点：碎片最小化

缺点：产生大量微小碎片（每次剩一点点）

408最常考

🥉 最坏适应 WF

排序：按容量递减

策略：找最大的分区

优点：不易产生微小碎片

缺点：大分区被迅速消耗

🔁 邻近适应 NF

排序：按地址（循环链表）

策略：从上一次结束位置继续查找

优点：均衡分布，查找效率高

缺点：缺乏大分区

4️⃣ 基于索引搜索的分配算法 ⭐重点补充

当系统规模很大时，空闲分区链很长，顺序搜索效率低。索引搜索将空闲分区按大小分类，用索引表管理，避免遍历整个内存。

快速适应特点

  • 分类：按进程常用的空间大小分类（2KB、4KB、8KB...）
  • 分配：查索引表 → 定位链表 → 取第一块（不分割！）
  • 优点：查找效率高、不产生分割碎片（无外部碎片）
  • 缺点：回收合并复杂、系统开销大、可能有内部碎片

伙伴系统核心规则

  • 大小规则：所有分区大小 = 2^k（1 ≤ k ≤ m）
  • 分配：找满足 2^(i-1) < n ≤ 2^i 的最小块；若无则分裂更大的块
  • 伙伴地址：若 (x/2^k) 为偶数 → 伙伴=x+2^k；为奇数 → 伙伴=x-2^k
  • 回收：检查伙伴是否空闲 → 是则合并为 2^(k+1) → 递归检查
  • 应用：Linux内核物理内存管理

🎯 伙伴系统真题示例

请求分配100KB（约25页），最小单元1页：

  1. 2^4=16 < 25 ≤ 32=2^5，需要分配 32页
  2. 若32页链表为空 → 找64页块
  3. 64页块 分裂为两个32页伙伴
  4. 一个32页 分配，另一个入32页链表

🔢 哈希算法（Hash-based）

哈希算法特点

  • 结构：哈希表（关键字=分区大小）+ 链表
  • 分配：Hash(size) → 定位表项 → 取链表第一块
  • 优点：查找极快，平均 O(1)
  • 缺点：哈希冲突处理复杂、回收合并困难

📖 基本原理

  • 紧凑技术：移动所有已分配分区，使它们相邻，合并出大的连续空闲区
  • 前提：作业在内存中可以移动（动态运行时装入）
  • 代价：需要大量时间复制数据；需暂停作业执行

🔧 硬件支持：重定位寄存器

物理地址 = 逻辑地址 + 重定位寄存器值

关键理解：紧凑后只需修改各作业的重定位寄存器值，无需修改程序内部的逻辑地址！

✅ ❌ 特点

优点

  • 消除外部碎片
  • 提高内存利用率
  • 支持作业在内存中移动

缺点

  • 需要硬件支持（重定位寄存器）
  • 紧凑开销大（I/O时间）
  • 不能频繁紧凑
